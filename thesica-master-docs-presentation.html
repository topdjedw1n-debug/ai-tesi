01

Що описували старі docs

Документи описують повний B2C self-serve продукт: студент заходить на сайт, створює документ, платить за сторінки, чекає генерацію, експортує результат, а адміністратор керує користувачами, платежами, refund-запитами і системними налаштуваннями.

Product
Full doc

Фокус не на окремих секціях, а на генерації повного академічного документа.

Pricing
€0.50

Pay-per-page модель, мінімум 3 сторінки, максимум 200 сторінок, валюта тільки EUR.

Stack
Next/FastAPI

Next.js 14 frontend, FastAPI backend, PostgreSQL, Redis, MinIO, OpenAI/Anthropic.

Важливий контекст

Це старий опис TesiGo. Він корисний як карта вже продуманих модулів, але не весь текст треба переносити буквально в поточний Thesica-напрям. Зараз бренд і стратегія зміщені до Scholarly Press і QA-first legitimacy.

02

Продукт і бізнес-модель

Старі docs формулюють продукт як AI-powered платформу для дипломних, курсових і дисертацій з гарантією унікальності, багатомовністю, експортом і автоматизованим вибором AI pipeline.

Що продаємоcore
  • Генерація повного академічного документа, не лише outline.
  • Підтримка типів робіт: дипломні, курсові, дисертації.
  • Експорт у DOCX/PDF без пост-редагування після генерації.
  • Real-time progress і background generation.
  • Автоматичний вибір AI provider/model системою.
Як монетизуємоpayment
  • Pay-per-page: базово €0.50 за сторінку.
  • Мінімальний чек: 3 сторінки = €1.50.
  • Максимальний документ: 200 сторінок.
  • Stripe для checkout, history і refund flow.
  • Ціну можна змінювати через admin pricing settings.

03

Користувацький шлях

UX-док описує класичний conversion funnel: discovery → landing → signup → dashboard → document creation → payment → generation → export → repeat usage.

Discovery і landing

Користувач приходить з Google/social, бачить hero, фічі, how it works, pricing і чіткий CTA.

Magic link auth

Email-only login, одноразовий токен на 15 хвилин, rate limit 3 magic links на день.

Dashboard

Центральний хаб: stats overview, create document form, documents list, recent activity.

Create document

Title, topic, language, target pages, additional requirements і optional file upload.

Price preview і payment

Користувач бачить формулу вартості: сторінки × €0.50, платить через Stripe.

Background generation

Після payment success запускається background job з WebSocket progress updates.

Completed document

Користувач бачить повний текст, stats, export DOCX/PDF, regenerate/delete/refund actions.

Repeat usage

Payment history, profile stats і document list мають вести до повторного створення робіт.

Архітектура системи

Master-док описує стандартну production web architecture: Next.js frontend, FastAPI backend, PostgreSQL persistence, Redis для sessions/cache, MinIO для storage, AI APIs для generation.

Design decisionsarchitecture
  • Async everywhere for I/O operations.
  • Strict typing through Pydantic.
  • Stateless app sessions stored outside memory.
  • Streaming for large files and long generation.
  • Background jobs for document generation.
Core tablesdata
  • users: identity, email, active state.
  • documents: owner, title, content, status, pages.
  • payments: amount, status, Stripe intent.
  • jobs: status, progress, errors.
  • audit_logs and pricing_config.
06

AI pipeline

Старий pipeline уже містив правильну ідею: не просто виклик LLM, а research, outline, section generation, quality assurance, export, token tracking і fallback/retry strategy.

Input processing

Validate requirements, estimate costs, check user balance or payment state.

Source research / RAG

Search via Perplexity/Tavily/Serper, retrieve academic papers, format citations, build context.

Outline generation

Create structure, define sections, allocate pages across the document.

Content generation

Generate by sections, include RAG sources, save checkpoints, stream to storage, clear memory.

Quality assurance

Grammar check, plagiarism check, formatting validation, eventual AI-risk checks.

Delivery

Export to DOCX/PDF, store in MinIO, notify user and update document status.

Retry
2→32s

Exponential backoff: 2, 4, 8, 16, 32 seconds.

Fallback
4 models

gpt-4 → gpt-4-turbo → gpt-3.5-turbo → claude-3.5-sonnet.

Checkpoint
5 min

Checkpoint interval planned around long-running document generation.

Self-learning concept

Docs also planned a self-learning loop: collect high-quality documents, filter by quality/plagiarism/AI-detection/user satisfaction, export JSONL for fine-tuning, retrain monthly, A/B test on 10% traffic, deploy only after quality lift. Status was planned after 100+ successful documents.

08

Оплата, checkout і refunds

Старий UX робив оплату обов'язковим переходом перед full generation: document → price breakdown → Stripe → payment success → background generation.

Payment flowStripe
  • Payment page shows title, pages, language.
  • Breakdown: pages × €0.50 = total.
  • Stripe Elements validates card in real time.
  • Success verifies session_id and starts generation.
  • Cancel page offers retry or return to dashboard.
Refund flowcontrolled
  • User can request refund from failed/completed document.
  • Admin approves or rejects request.
  • Technical system error can trigger automatic refund.
  • No automatic cancellation after successful payment.
  • Payment history shows completed, failed, refunded statuses.

10

Безпека і compliance

Документи вже виділяли критичні security gaps: IDOR, weak JWT keys, file validation, backups, webhook signatures. Для академічного/платіжного продукту це не косметика, а launch blocker.

Implemented / designedbase
  • Magic link auth with JWT.
  • Refresh tokens and Redis session support.
  • Rate limits: IP, user, email magic links.
  • Security headers: CSP, X-Frame-Options, nosniff, HSTS.
  • GDPR concepts: right to be forgotten, data export, retention.
Launch blockersmust fix
  • IDOR ownership checks for documents and user resources.
  • Strong 32+ char secrets for JWT/session keys.
  • Magic bytes validation for uploaded files.
  • Encrypted 3-2-1 backups for PostgreSQL.
  • Stripe webhook signature verification.
Security interpretation

Старий master-док чесно називав ці проблеми критичними. У презентації це треба залишити як mature сигнал: продукт не оголошується production-ready, поки user data, payments і document ownership не захищені.

Monitoring, debugging і operational controls

Master-док уже передбачав production observability: structured JSON logs, correlation IDs, Prometheus metrics, Sentry, critical alerts і health endpoints.

Logs
  • Structured JSON logging.
  • Correlation ID.
  • User/action/details fields.
  • logs/app.log and audit logs.
Metrics
  • HTTP request count/duration.
  • Generation duration.
  • AI API calls and costs.
  • Prometheus port 9090.
Alerts
  • API error rate > 5%.
  • p95 response > 2s.
  • Memory > 85%.
  • Disk < 10GB.
  • AI failures > 3 in row.
Debugging model

Docs include debug endpoints for dev-only config/cache/jobs checks and common failure explanations: memory leak after large documents, slow generation due to AI API limits, payment failures through Stripe webhooks.

Roadmap, technical debt і що з цього брати далі

Старий roadmap має корисну структуру, але його треба перевести з generic self-serve SaaS у поточний Thesica/QA-first контекст.

Immediate before launchblockers
  • Fix IDOR and authorization boundaries.
  • Harden JWT and secrets.
  • Add file validation and backups.
  • Integrate background job service.
  • Add webhook signature verification.
Technical debtknown
  • MyPy issues around ORM, annotations, decorators.
  • Low coverage in admin/humanizer/background jobs modules.
  • Overall coverage noted around 44%, target 80%.
  • No MyPy CI gate and no coverage threshold enforcement.
  • Specific bugs in rate limit and exception typing.

A/B testingImprove model/output quality.Useful after a stable eval set, not before quality proof.
Mobile appsLong-term user convenience.Not priority until internal production flow is proven.

14

Висновок

Ці старі docs не треба викидати: вони містять велику карту системи. Але їх треба читати як історичний self-serve blueprint, який зараз треба підпорядкувати Thesica-стратегії.

Старий TesiGo описував повну SaaS-машину: користувач, оплата, генерація, export, admin, monitoring. Поточна Thesica має взяти цю інженерну карту, але поставити в центр не checkout, а доказ якості і release-рішення.

Synthesis from MASTER_DOCUMENT + QUICK_START + UX Structure
Що лишити

Архітектуру, auth, document states, background jobs, export, admin controls, monitoring, security discipline.

Що переосмислити

Self-serve checkout як перший крок, generic landing claims, користувацький direct-to-delivery flow.

Що додати

QA-first release gate, citation/claim evidence, editor routing, human minutes, outcome learning loop.
