1. Головне рішення

Фронтенд не має бути одним великим кабінетом. Його треба розділити на чотири поверхні, кожна з окремою відповідальністю.

Public

Публічний сайт

Маркетинг, lead capture, FAQ, terms, privacy. Без обіцянок обходу AI/plagiarism detection.

Client

Клієнтський портал

Заявка, quote/payment, безпечний статус, delivery files, revision request.

Manager

Manager/Admin console

Production control, QA evidence, release gate, payments, refunds, users, audit.

Editor

Editor workspace

Вузька черга задач із конкретними findings, а не повна ручна перечитка кожної роботи.

Done: менеджер проводить реальне замовлення від intake до release decision без бази, логів і допомоги розробника. Клієнт бачить тільки безпечний статус і релізнуті файли.

2. Ролі та права

Ролі потрібні не для косметики, а щоб client, editor і manager не бачили зайвого і не могли обійти QA-first процес.

РольБачитьНе може
adminКористувачі, ролі, settings, payments, refunds, всі cases, audit, overrides.Не має бачити raw secrets або приватні provider values.
managerIntake, jobs, QA evidence, editor tasks, delivery, minutes, release decision.Не змінює system settings без окремого права.
editorAssigned tasks, секції, findings, comments, QA rerun result.Не бачить payments, refunds, global settings, release override.
clientСвої заявки, safe status, uploaded files, released deliverables, messages.Не бачить raw AI output до release approval і internal QA details.
supportOrders, clients, status, refund request intake.Не робить release override і не змінює global settings.

3. Data model direction

Поточний `Document` не треба ламати. Додаємо production layer навколо нього, щоб бізнес-замовлення не змішувалось із raw document state.

Core

ProductionCase

Business order: client, document, manager, editor, deadline, citation style, requirements, human minutes, release status.

Gate

ReleaseGateResult

Один gate result: key, severity, status, evidence, provider, override policy, override reason.

Editor

EditorTask

Finding-specific задача з section, assigned editor, status, resolution notes і minutes spent.

Audit

AuditEvent

Actor, action, target, reason, timestamp, correlation id для release, override, retry, assignment.

4. Product surfaces

Кожна поверхня має один job-to-be-done. Це зменшує ризик, що client portal випадково стане внутрішньою адмінкою.

Admin dashboard

Open cases, due in 48h, stuck jobs, failed QA gates, waiting editor, ready release, token cost, refund alerts.

Production case detail

Tabs: Overview, Intake, Generation, QA Evidence, Editor Tasks, Payments, Delivery, Audit.

Editor workspace

My tasks, task detail, section editor, QA finding context, submit resolution.

Client portal

Request received, Requirements review, Drafting, Quality review, Editorial review, Delivered.

Release gate

Release disabled when blocking gates fail or have no data without valid override.

Public website

Allowed: AI-assisted academic drafting with human quality controls. Disallowed: bypass promises.

5. Фази реалізації

Порядок важливий. Не починаємо з публічного сайту, поки production case і release gate нечіткі.

A

Align current admin around QA-first

Перефокусувати поточну адмінку з vanity metrics на production risk.

Exit: document detail показує QA evidence, provenance і чесні no-data стани.
B

Production case layer

Додати business-order шар навколо Document, окремі статуси і admin list/detail.

Exit: manager створює case без Stripe і без ручної роботи в базі.
C

Release gate contract

Gate config, gate results, override policy, audit, release endpoint.

Exit: failed/no-data blocking gate блокує release без валідного override.
D

Editor workspace

QA findings стають конкретними editor tasks із minutes tracking.

Exit: editor закриває finding, manager бачить task status.
E

Client-safe portal

Клієнт бачить safe status і delivery тільки після release approval.

Exit: client не може завантажити unreleased files або запустити uncontrolled generation.
F

Public website and sales

Публічна упаковка, pricing/request quote, payment wiring, FAQ, terms.

Exit: new request входить у production flow і не обходить QA.

8. Ризики та guardrails

Risk

Self-serve занадто рано

Guardrail: client portal стартує тільки з safe status; raw output не показується до release.

Risk

`document.status` перевантажений

Guardrail: окремі production status dimensions.

Risk

No data виглядає як success

Guardrail: `no_data` є first-class state і блокує release для blocking gates.

Risk

Editor переписує все вручну

Guardrail: задачі мають бути finding-specific, а minutes spent трекаються.

Stop condition: якщо два implementation attempts не покращують primary signal, зупинитися й переформулювати owning problem.

1. Core decision

The frontend must not become one overloaded cabinet. Split it into four surfaces with separate responsibilities.

Public

Public website

Marketing, lead capture, FAQ, terms, privacy. No AI/plagiarism bypass promises.

Client

Client portal

Request, quote/payment, safe status, delivery files, revision request.

Manager

Manager/Admin console

Production control, QA evidence, release gate, payments, refunds, users, audit.

Editor

Editor workspace

A focused task queue from QA findings, not a full manual reread of every paper.

Done: a manager can run a real order from intake to release decision without database tools, logs, or developer help. The client sees only safe status and released files.

2. Roles and permissions

Roles keep the client, editor, and manager from seeing too much or bypassing the QA-first process.

RoleMust seeMust not do
adminUsers, roles, settings, payments, refunds, all cases, audit, overrides.No raw secrets or private provider values.
managerIntake, jobs, QA evidence, editor tasks, delivery, minutes, release decision.No system settings unless explicitly allowed.
editorAssigned tasks, sections, findings, comments, QA rerun result.No payments, refunds, global settings, release override.
clientOwn requests, safe status, uploaded files, released deliverables, messages.No raw AI output before release approval, no internal QA details.
supportOrders, clients, status, refund request intake.No release override, no global settings.

3. Data model direction

Do not break the current `Document` model. Add a production layer around it so the business order is not mixed with raw document state.

Core

ProductionCase

Business order: client, document, manager, editor, deadline, citation style, requirements, human minutes, release status.

Gate

ReleaseGateResult

One gate result: key, severity, status, evidence, provider, override policy, override reason.

Editor

EditorTask

A finding-specific task with section, assigned editor, status, resolution notes, and minutes spent.

Audit

AuditEvent

Actor, action, target, reason, timestamp, correlation id for release, override, retry, assignment.

5. Implementation phases

Order matters. Do not start with the public website while production case and release gate are unclear.

A

Align current admin around QA-first

Move current admin from vanity metrics to production risk.

Exit: document detail shows QA evidence, provenance, and honest no-data states.
B

Production case layer

Add a business-order layer around Document, separate statuses, and admin list/detail.

Exit: manager creates a case without Stripe or manual database work.
C

Release gate contract

Gate config, gate results, override policy, audit, release endpoint.

Exit: failed/no-data blocking gate blocks release without valid override.
D

Editor workspace

QA findings become concrete editor tasks with minutes tracking.

Exit: editor closes a finding, manager sees task status.
E

Client-safe portal

Client sees safe status and delivery only after release approval.

Exit: client cannot download unreleased files or trigger uncontrolled generation.
F

Public website and sales

Public packaging, pricing/request quote, payment wiring, FAQ, terms.

Exit: new request enters production flow and cannot bypass QA.
